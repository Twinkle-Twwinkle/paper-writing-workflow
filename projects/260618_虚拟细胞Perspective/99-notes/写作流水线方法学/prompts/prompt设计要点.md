# Prompt 设计要点（为什么这两个 prompt 这么写）

提炼可复用技能时，这些是 prompt 里**真正起作用、值得保留**的结构性手法。与具体题目无关，可迁移。

## 通用心法

1. **"脚手架 + 反驳授权"**（P3 用得最足）
   主控先预读关键几行、形成一个**假设性框架**，把它交给高推理 agent，但**明确写"这是待你验证的脚手架，不是结论，发现不符就推翻"**。
   - 好处：agent 不从零开始（省 token、抓得准），又不被框死（保留推翻空间）。
   - 反模式：把假设当结论塞给 agent → 它会一路附和，证据修正不了假设。

2. **"证据修正假设"写进核心原则**
   反复强调 context/初稿只是**初始假设**，必须被真实证据 confirm / sharpen / overclaim / insufficient 四档裁决。这是整条流水线的灵魂：**不是用文献给预设观点背书，是让文献改观点**。

3. **给"抗灌水"的硬判据**
   明确"判断支持强度看**支持论文数（distinct papers）**，不看触点数（touchpoints）"。因为单篇论文可能对一条 claim 堆 15 个触点。任何"按计数排序"的步骤都要先想清楚**计什么单位**。

4. **逐字保真分级**
   - P3（观点步）：允许 agent 释义、压缩，引文是"弹药示意"——**但事后会被 P4 反审**。
   - P4（成骨步）：逐字保真是**硬约束**，要求"用 python 从数据源拉原句、禁手工转写、找不到就标 UNBACKED 不许编造"。
   - 这个分级让"敢下判断"和"可被审计"两个目标不打架：先放开想，再绑死证。

5. **方向借用要诚实括注**
   一条 `challenges`/`nuances` 方向的证据，可能恰恰支持你的论点（如 workaround 工具的"自承局限"=缺口铁证）。允许借用，但**强制在该条后中文括注理由**，让审计能判断"借用是否诚实"而非偷换。

6. **让下游 agent 反审上游**（P4 反审 P3）
   P4 被要求"P3 引的句子在数据源找不到就标 UNBACKED"。结果真的逮出 P3 有 6 处释义/拼接非逐字。**把校验职责下放给下一棒**，比自己回头查更省力，也更可信。

7. **schema 强制结构化**（精练 4.0 / build_matrix 用）
   能用 JSON schema 把输出钉死的就钉死（excerpt-centric、每条原句自带 touchpoints）。agent 自由发挥的字段越少，下游确定性代码越好接。
   - **坑**：去掉枚举约束后，41 个 agent 会把同一个 claimId 翻译成 60 种写法（见决策日志"claimId 碎裂"）。schema 自由度与一致性要权衡。

## 模型 / 派发

- P3、P4 都用 **opus**（推理 + 写作质量优先）。
- 一个会话里**能用确定性代码做的别派 agent**（P2 矩阵、P5 审计都是纯 python）。agent 只用在"需要判断/读海量文本/写作"的步骤。
- 语气微调（"收一点"）这类活，**主控直接写比派 agent 稳**——编译这一步主控自己干。
