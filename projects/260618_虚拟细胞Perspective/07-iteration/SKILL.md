# 论文迭代（Paper Iteration Module）

> **定位**：现有 `00→06` 流水线解决「从无到有写出初稿」；本模块接在其后，解决「从有到更好——拿到新反馈后稳定地摇出下一版」。设计目标是**可重复跑很多轮且不出乱子**。
>
> **一句话**：把「当前稿 + 一条新 context」喂进去，按固定纪律摇出「改好的下一版 + 一份说明为什么这么改的迭代日志」。

---

## 为什么要这个模块（要防的三类事故）

迭代轮数一多，最容易出三种事故。本模块每一条纪律都对应防住其中之一：

1. **改崩**——改着改着把之前审过的论断改坏了（引文对不上、逻辑断了）。
2. **走样**——新意见没被忠实记录，过两轮就忘了导师到底要什么。
3. **版本糊**——分不清这是第几版、为什么这么改、上一版长什么样。

---

## 核心约束（整个迭代过程全程遵守）

1. **新 context 先保真、后提炼**：拿到新输入，**先逐字原话存档**（不改写、不裁剪、标注来源与日期），再另起一段提炼要点。原话区与提炼区物理分开。（防「走样」）
2. **地基不动、增量追加**：原始地基 context（如 `context.md`）**只读不改**；所有新意见以**补充文件**形式追加。需要时在地基之上分层（v1 地基 / v2 增量 / 本轮意见汇总），不就地覆盖旧 context。
3. **先 triage、不大拆大建**：动手前必须先做影响评估——这条意见**砸到哪几条 claim、哪几段**，定级为「仅补引文 / 加一段 / 改主线 / 仅措辞」。改动范围以评估结论为准，**默认最小改动**。
4. **先骨架后成文**：一切改动**先落在 `outline.md`（可审计骨架）**，每条论断仍绑 `[引用号] + 逐字原句`；骨架自洽后再由它**重编 `draft.md`**。**禁止绕过 outline 直接改 draft。**
5. **引文纪律，零编造**：每条**新增/改动**的论断都必须有逐字引文背书。**证据不足时不许硬写**——回头跑一小段 `02→05` 补语料（见 Step 3），把新 excerpt 收进 `05-distilled` 索引后再用。
6. **版本快照不可变**：上一版 `outline.md`/`draft.md` 视为**只读快照**，不就地覆盖。每轮产出一个**新版本号**，旧版本永久留痕可回溯。（防「版本糊」）
7. **全程留痕**：每轮必须写一条 **changelog**：本轮输入了哪条 context → 触发了哪些 claim 的什么改动 → 新版本号 → 审计结论。没有 changelog 的迭代视为未完成。

---

## 输入（三项；第 3 项可选）

1. **当前稿** —— 当前版本的 `06-outline-and-draft/outline.md` + `draft.md`（基线）。
2. **一份「新 context」** —— 本轮的迭代驱动，类型之一：
   - **PI / 导师意见**（如郭天南 VC≠DT、AIVC≠scRNA single-cell）；
   - **reviewer / 审稿意见**；
   - **新补充文献的要点**（来自一轮新的 `02→05`，如本项目的 `V2补充论文`）；
   - **自查发现的逻辑漏洞 / 论证缺口**。
3. **（可选）grounding 索引** —— 项目地基 context（thesis + claim 路线图）与 `05-distilled` 证据索引（`cards.json` / `evidence-matrix.md` / `citation-registry.md`），用于核对 claim 编号与逐字引文。若本轮改动触及引文，必须读它；改不动引文的纯措辞轮可跳过。

> 若用户只给了「新意见」却没指明当前是哪一版稿子，**先问清基线版本**，不要凭记忆假设。

---

## 输出

1. **新版本** `outline.md` + `draft.md`（按 §版本管理 落到一个新版本号下，旧版只读保留）。
2. **一条 changelog** 追加进 `07-iteration/迭代日志.md`（格式见文末模板）。
3. **审计结论**：逐字回搜的 PASS/不通过说明（接在 changelog 条目里）。

---

## Workflow（单轮迭代 = 七步）

### Step 0 — 锁定基线
确认本轮从哪一版改起（读当前 `outline.md` + `draft.md`）。记下基线版本号，作为 changelog 的 `from` 字段。

### Step 1 — 接收 & 保真新 context（防走样）
- 把新输入**逐字原话**存档到 `00-context/`（或本项目既有的「意见汇总」文件），标注**来源、日期、类型**。**不改写原话。**
- 另起一段**提炼要点**（你对原话的理解），与原话区物理分开。
- 本项目已有范本：`00-context/郭天南新增意见汇总_260622.md`（原话区 + 要点提炼区分层）——照此办理。

### Step 2 — 影响评估 / triage（防改崩的第一道闸）
对照地基 context 的 **claim 路线图**，判断这条新 context：
- **砸到哪几条 claim、哪几段**（逐条列出 claim 编号 / 段落）；
- **每处定级**：`仅补引文` / `加一段` / `改主线` / `仅措辞`；
- 产出一张**改动清单**（哪段、改成什么意思、为什么）。这张清单是后续所有改动的依据，**不在清单内的不改**。

### Step 3 — 必要时补语料（守住引文纪律）
若改动清单里出现「现有引文撑不住的新论断」：
- 回头跑一小段 `02→05`（下载→清洗→精练），把新文献的**逐字 excerpt** 收进 `05-distilled` 索引（本项目 `V2补充论文` 即此先例）；
- **没补到证据的论断，退回 Step 2 降级或删除**，不许硬写。

### Step 4 — 改 outline（先动骨架）
在 `outline.md` 上按改动清单加 / 改论断：
- 每条新论断仍绑 `[引用号] + 逐字原句`；
- **显式标注本轮的新增/修改**（如 `【v2 新增】`），方便审计和回溯；
- 先让骨架层逻辑与引文自洽，再进下一步。

### Step 5 — 重编 draft（先骨架后成文）
由改后的 outline **重新编译** `draft.md`：
- 保持期刊约束（字数上限、参考文献条数、article type）；
- 改动落地为连贯正文，不留半句无引文支撑的话；
- **不直接在 draft 上改意思**——意思的改动只能来自 outline。

### Step 6 — 审计 + 落版本 + 记 changelog（防改崩 + 留痕）
- **逐字回搜审计**：抽查每条引用是否能在 `04-markdown-clean` 原文里逐字命中，零编造、零改写；顺带检查旧版已审过的论断**没被本轮改坏**。
- **落版本**：按 §版本管理 生成新版本号，旧版只读保留。
- **写 changelog**：追加一条到 `07-iteration/迭代日志.md`（模板见文末）。审计不通过则不进版本号，回到对应 Step 修。

## 版本管理（很多轮的基础设施，二选一，全程一致）

**方案 A·文件夹快照（推荐给不熟 git 的同学）**
```
06-outline-and-draft/
  v1/ outline.md  draft.md      # 只读快照
  v2/ outline.md  draft.md      # 本轮产出
  latest -> v2/ (或在 07 日志里注明 latest)
07-iteration/
  迭代日志.md                    # 每轮一条 changelog
```

**方案 B·git（推荐给用 git 的同学）**
- 每轮一个 commit，并打 tag：`draft-v1` / `draft-v2` …；
- changelog 仍写进 `07-iteration/迭代日志.md`，与 tag 一一对应。

> 规则：**同一项目只用一种方案，不混用。** 旧版本一旦定版即不可变。

---

## 迭代日志模板（`07-iteration/迭代日志.md` 每轮追加一条）

```markdown
## v{N}  ·  {日期}  ·  from v{N-1}
- **输入 context**：{类型：PI意见/reviewer/新文献/自查} —— {一句话 + 原话存档文件链接}
- **triage 改动清单**：
  - {claim/段落} —— {定级：补引文/加段/改主线/措辞} —— {改成什么意思}
- **是否补语料**：{否 / 是：补了哪几篇，新 excerpt 落在 05-distilled 何处}
- **outline 改动**：{标了 【v{N} 新增/修改】 的条目编号}
- **draft 改动**：{受影响的章节/段落 + 字数变化}
- **审计结论**：{PASS / 不通过原因} —— 逐字回搜 {抽查了哪几条引用}
- **版本产物**：{文件夹快照路径 或 git tag}
```

## 边界（这个 skill 不做什么）

- **不负责从零写初稿**——那是 `00→06` 主流程的事；本模块只在已有 draft 之上迭代。
- **不替你判断意见对错**——导师/reviewer 说什么就忠实纳入并落实；你能做的是评估它砸到哪、需要哪些证据、提醒证据缺口。
- **不下载/清洗/精练**——需要补语料时**调用**已有的 `论文下载`/`论文清洗`/`精练` skill，本模块只负责把结果接回稿子。
- **不擅自改地基 context**——地基只读，新意见一律增量追加。
